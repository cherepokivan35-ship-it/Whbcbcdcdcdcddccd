# Бесплатные варианты для RNS/Reticulum в 2026

Коротко: для настоящего аналога `rns.moskow` нужен **постоянный процесс** и **входящий TCP-порт**. Бесплатные веб-хостинги часто дают только HTTP-сайт, поэтому для RNS они не подходят напрямую.

## Вариант 1 — самый реалистичный без карты: свой ПК/ноутбук + Pinggy TCP tunnel

Это не VPS, но это реально бесплатный способ получить публичный `host:port` без белого IP и без банковской карты.

1. Установите Python и Reticulum:
   ```bash
   python3 -m pip install --user rns
   ```
2. Запустите RNS локально из этого проекта:
   ```bash
   ./scripts/run-local.sh
   ```
3. Во втором терминале запустите TCP-туннель:
   ```bash
   ./scripts/tunnel-pinggy.sh 4242
   ```
4. Pinggy напечатает публичный адрес и порт. Клиентам надо указать этот `host` и `port` в `TCPClientInterface`.

Минусы бесплатного туннеля: адрес/порт обычно временные, терминал должен быть открыт, стабильный зарезервированный адрес обычно платный.

## Вариант 2 — домашний интернет с белым IP или IPv6

Если провайдер даёт белый IPv4 или нормальный входящий IPv6:

1. Запустите проект через Docker Compose или `./scripts/run-local.sh`.
2. Откройте/пробросьте порт `4242/tcp` на роутере.
3. Клиентам дайте `ваш_IP:4242`.

Это самый близкий бесплатный вариант к настоящему серверу, но работает только если входящие соединения не заблокированы провайдером.

## Вариант 3 — Koyeb / похожие PaaS с Docker

Если сервис доступен из вашей страны и даёт бесплатный инстанс без подходящей карты, можно попробовать Docker-деплой из этого репозитория. Важно: проверьте, что тариф не засыпает и умеет прокидывать именно TCP, а не только HTTP.

Общий алгоритм:

1. Залить репозиторий на GitHub.
2. Создать service из Dockerfile.
3. Указать порт приложения `4242`.
4. Если панель требует route/protocol, выбрать TCP route/listener, если он доступен.

## Почему Vercel, GitHub Pages, обычный сайт и Express.js не решают задачу

RNS `TCPServerInterface` — это не HTTP endpoint. Vercel/Pages могут отлично хостить сайты, но они не дают постоянный произвольный TCP listener `0.0.0.0:4242`. Express.js тоже HTTP-сервер; он не превращает serverless-хостинг в постоянный Reticulum transport.

## PythonAnywhere

PythonAnywhere полезен для Python-сайтов и консольных скриптов, но бесплатный аккаунт обычно ограничен веб-приложением/консолями и не является нормальным публичным TCP-сервером для RNS. Можно экспериментировать с исходящим клиентом Reticulum, но как публичный `IP:4242` для других — это плохой вариант.

## Playit.gg

Раньше его часто советовали как бесплатный туннель, но сейчас произвольные TCP-туннели могут быть платной функцией. Для RNS нужен именно TCP, поэтому перед настройкой проверьте текущие ограничения аккаунта.

## Честный вывод

Если совсем без карты и без VPS, самый рабочий вариант — **запустить RNS у себя и вынести порт наружу через бесплатный временный TCP-туннель Pinggy**. Это не идеально, но это уже не скелет: проект можно запустить, а клиентам дать полученный `host:port`.
